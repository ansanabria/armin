# Generated migrations by default, manual SQL by exception

Armin treats `schema.ts` as the canonical current database shape and starts ordinary schema changes with Drizzle-generated migrations, but allows hand-authored or materially hand-edited SQL when the upgrade semantics matter more than generated DDL: data backfills, table renames, SQLite rebuild constraints, and preservation of user history. We chose this over fully manual migrations because Drizzle keeps simple DDL aligned with the typed schema, and over generated-only migrations because local-first profile databases need safe upgrade paths across real persisted data.

Once a migration has appeared in any tagged release artifact, including alpha, it must not be rewritten to repair or simplify history; a new migration carries the fix forward. Before a tagged release, rewriting the newest local migration is acceptable only when it has not become someone else's durable upgrade path. The existing migration history has incomplete Drizzle snapshot metadata, so we do not try to reconstruct old snapshots; future Drizzle-generated migrations should commit the generated snapshot and journal changes alongside the SQL. Manual or materially edited migrations should be verified against an old-shape temporary database before release, but those checks should not become permanent tests unless they protect a long-lived core service invariant rather than a one-time upgrade path.
