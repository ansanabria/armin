# Generated migrations by default, manual SQL by exception

Armin treats `schema.ts` as the canonical current database shape and uses Drizzle-generated migrations for ordinary schema changes. Hand-authored or materially hand-edited SQL is an exception that must be discussed before editing the migration, and is reserved for upgrade semantics Drizzle cannot infer safely: data backfills, table renames, SQLite table rebuild constraints, and preservation of user history. We chose this over fully manual migrations because Drizzle keeps simple DDL aligned with the typed schema, and over generated-only migrations because local-first profile databases may still need safe upgrade paths across real persisted data.

Before production use, the migration history was reset to one generated `initial_schema` migration so the database starts from a uniform clean slate. Once a migration has become someone else's durable upgrade path, it must not be rewritten to repair or simplify history; a new migration carries the fix forward. Future Drizzle-generated migrations should commit the generated snapshot and journal changes alongside the SQL. Manual or materially edited migrations should be verified against an old-shape temporary database before release, but those checks should not become permanent tests unless they protect a long-lived core service invariant rather than a one-time upgrade path.
